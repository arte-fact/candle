# Post-P1 Phase 2c bench — widened MMQ tile

**Date**: 2026-04-11
**Host**: 3× AMD MI50 (gfx906), ROCm 7.1.1
**Model**: `Qwen3.5-9B-Q4_1.gguf`, 1 GPU, 1149-tok prompt + 128 decode
**Change**: commit `6ff79d62` — `mul_mat_q*_gfx906_v2` TILE_N widened from 8 → 32

## Throughput progression

| Stage | Prefill (pp1149) | Decode (tg128) | Total GPU time | Gap vs turbo |
|---|---:|---:|---:|---:|
| Pre-P0 baseline (BENCH-3WAY) | 78.60 t/s | 35.78 t/s | 17.4 s | 6.5× |
| Post-P0 Phase 1+2 | ~145 t/s | ~35.4 t/s | 11.4 s | 4.2× |
| **Post-P1 Phase 2c (this)** | **~195 t/s** | **~36.2 t/s** | **9.4 s** | **3.5×** |
| Turbo reference | 450 t/s | 58.76 t/s | 2.7 s | 1× |

**Cumulative**: 2.48× prefill, 1.02× decode vs pre-P0.
**This commit alone**: +34 % prefill (145 → 195 t/s), −17 % total GPU time.

Only the MMQ category moved, and per-call on the MMQ kernel halved:

| MMQ kernel | calls | avg µs | total ms |
|---|---:|---:|---:|
| `mul_mat_q4_1_gfx906_v2` (TILE_N=8, pre-P1) | 128 | 31,473 | 4028 |
| **`mul_mat_q4_1_gfx906_v2_tile32` (post-P1)** | **128** | **15,782** | **2020** |
| Turbo's `mul_mat_q<Q4_1, mmq_x=64>` reference | 128 | ~2,500 | ~320 |

One Wave64 warp per block with TILE_N=32 sits at **9.6 warps/CU**
(576 blocks / 60 CUs) — the sweet spot where the 1149×1024 projection
just saturates gfx906 without dropping occupancy. TILE_N=64 was tried
and regressed to **141 t/s prefill** (−4 % vs TILE_N=8) because it
shrinks the grid to 2.7 warps/CU, below the latency-hiding floor.

## What this commit does NOT touch

The remaining 6.3× per-call gap (15.8 ms vs turbo's 2.5 ms) lives in
optimisations that all require a structural kernel rewrite:

- **No LDS staging** of X or Y tiles — both still come through VGPRs
  and L1 broadcast. Turbo's inner loop computes from LDS exclusively.
- **No multi-warp blocks** — still one Wave64 per tile. Turbo uses
  2 warps per block (128 threads) on gfx906.
- **No vectorised int4 Y loads** via inline `global_load_dwordx4` asm.
- **No async prefetch / double buffering** of the next K-tile.
- **No stream-K decomposition** — irrelevant for our shapes (we have
  enough blocks to saturate 60 CUs), mattered more for turbo's small-
  M cases.

Each of those is a ~200-500 LOC change and would land as Phase 2d /
2e / etc. The low-hanging fruit in this commit was a single template
parameter change.

Qwen35-9B's output head tensor is Q5_K, so every decode token hits
the `mul_mat_vec_q5_K_q8_1_cuda8` chunked fallback. Extending MMQ to
cover K-quants (roadmap P5) would drop this 3280 ms → ~400 ms, saving
another ~2.9 s. That's the biggest remaining lever.

## Recommended next steps

1. **P5 — K-quant MMQ extension**. Biggest single lever remaining
   (~2.9 s GPU time savings, ~+20 % decode). Mechanical extension of
   the Phase 2 MMQ template + the new tile widening.
2. **P1 Phase 2d/2e — LDS staging + vectorised Y loads for Q4_1 MMQ**.
   Closes the 6.3× per-call gap on the matmul path (~1.5 s savings),
   bigger kernel rewrite. Defer until after P5 since the absolute
   win is smaller.
3. **P4 — fused norm+silu+quantize**. ~300 ms GPU time savings.
   Small but free.

Projected post-P5 total GPU time: ~6.5 s → **~285 t/s prefill, ~55 t/s
decode**. Matching or exceeding turbo's 58 t/s decode.
